# Skills建设总计划 — 从框架到细节

## 统一架构

三个维度，不是三套独立系统：

```
Skills（能力）= 公司能做什么
  ↕
Roles（角色）= 谁来做
  ↕
Workflows（流程）= 怎么串起来自动跑
```

---

## 全局依赖图

```
                    ┌─────────────────┐
                    │  0. 公司知识库    │ ← 所有Skill都要读这个
                    │  company-context │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │ 1. 情报系统 │  │ 2. 内容引擎 │  │ 3. 软件工厂 │
     │ (信息输入)  │  │ (价值输出)  │  │ (产品交付)  │
     └──────┬─────┘  └──────┬─────┘  └──────┬─────┘
            │               │               │
            ▼               ▼               ▼
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │ 4. 分发系统 │  │ 5. 转化系统 │  │ 6. 运营中台 │
     │ (触达客户)  │  │ (客户成交)  │  │ (内部管理)  │
     └────────────┘  └────────────┘  └────────────┘
```

**建设原则：从上到下，从左到右。上面的不建好，下面的没法跑。**

## 建设顺序（6个阶段，每阶段有明确交付物）

### 阶段0：公司知识库（基座，所有Skill的前提）

**做什么：** 把公司所有稳定信息整合成一个可被其他Skill自动调用的知识库。

**具体建设内容：**

| Skill | 功能 | 数据来源 | 交付物 |
|-------|------|---------|--------|
| `company-context-manager` | 维护公司核心信息，回答"我们公司是做什么的" | company-context.md + memory中的业务线详情 | 一个YAML/JSON格式的结构化知识库 |

**包含的信息模块：**
```
company-knowledge/
├── business-lines/        ← 5条业务线详细定义（客户/痛点/卖点/盈利模式）
│   ├── A-transit.yaml
│   ├── B-returns.yaml
│   ├── C-software.yaml
│   ├── D-cross-border.yaml
│   └── E-ai-service.yaml
├── team/                  ← 团队结构（谁负责什么）
│   └── org-structure.yaml
├── tech-stack/            ← 技术栈（Java/Spring Boot/MySQL/Redis）
│   └── stack.yaml
├── customers/             ← 客户画像（4类客群）
│   └── customer-profiles.yaml
├── competitors/           ← 竞品档案（按业务线分）
│   └── competitor-map.yaml
├── brand/                 ← 品牌资产（Logo/色/Slogan/声线）
│   └── brand-assets.yaml
└── config/                ← 系统配置（API Key/飞书/工具）
    └── system-config.yaml
```

**为什么先做这个：** 后面建的每一个Skill，第一步都是"读取公司信息"。没有这个，其他Skill都是空中楼阁。

---

### 阶段1：情报系统（让公司"有眼睛"）

**做什么：** 每天自动获取行业信息，不再手动搜。

| Skill | 角色 | 功能 | 依赖 |
|-------|------|------|------|
| `intel-scanner` | 情报分析官 | 按5条业务线每日扫描全渠道 | 阶段0（知道每条线该搜什么） |
| `deep-crawler` | 情报分析官 | 对重点文章做全文提取+分析 | Firecrawl MCP |
| `competitor-tracker` | 竞品追踪员 | 每周爬竞品官网对比变化 | Firecrawl MCP |
| `social-monitor` | 情报分析官 | 监控16个社交平台热门内容 | agent-reach |

**自动化：** n8n工作流 — 每天06:00触发 → 输出情报简报 → 推飞书管理群

**交付物：** 每天一份按业务线分模块的情报简报（自动生成，自动推送）

---

### 阶段2：内容引擎（让公司"有嘴巴"）

**做什么：** 基于情报自动生产内容。

| Skill | 角色 | 功能 | 依赖 |
|-------|------|------|------|
| `content-planner` | 内容策略师 | 根据情报规划本周内容日历 | 阶段1（有情报才能规划） |
| `copywriter` | 内容创作员 | 按平台×风格×业务线写文案 | 阶段0（知道业务卖点） |
| `xhs-image-gen` | 内容创作员 | 小红书信息图生成 | baoyu-xhs-images |
| `video-producer` | 内容创作员 | 分镜脚本→图→视频→配音→合成 | Remotion + LibTV + CosyVoice |
| `style-manager` | 内容创作员 | 管理8种风格模板 | 模板库已建 |

**自动化：** n8n工作流 — 08:00接收情报 → 08:30规划内容 → 09:00生成内容

**交付物：** 每天自动产出多平台多风格的内容成品包

---

### 阶段3：软件工厂（让C线"能交活"）

**做什么：** 把软件开发标准化，一个人顶一个团队。

| Skill | 角色 | 功能 | 依赖 |
|-------|------|------|------|
| `requirement-analyzer` | 产品经理 | 客户需求→6个追问→PRD | gstack /office-hours |
| `prd-generator` | 产品经理 | 生成产品需求文档 | gstack /plan-ceo-review |
| `architecture-designer` | 架构师 | PRD→技术方案+API设计 | gstack /plan-eng-review |
| `dev-workflow` | 工程师 | 按方案写代码+测试 | Superpowers + gstack |
| `code-audit` | QA | 代码审查+自动测试 | gstack /review + /qa |
| `doc-generator` | 文档员 | 自动生成项目文档 | gstack /document-release |

**已有基础：** gstack 28个技能已安装，Superpowers已安装。只需要包装成我们公司标准流程。

**交付物：** 一套从需求到交付的标准开发SOP，彭涛和Rita都能用

---

### 阶段4：分发系统（让内容"到客户眼前"）

**做什么：** 内容自动发布到各平台。

| Skill | 角色 | 功能 | 依赖 |
|-------|------|------|------|
| `publisher-xhs` | 分发员 | 小红书发布 | apostify/Playwright |
| `publisher-wechat` | 分发员 | 公众号发布 | baoyu-post-to-wechat |
| `publisher-douyin` | 分发员 | 抖音发布（网页版） | apostify/Playwright |
| `schedule-manager` | 分发员 | 排期管理+错峰发布 | n8n |
| `platform-adapter` | 分发员 | 同一内容适配不同平台格式 | AI适配 |

**前提：** 需要先解决指纹浏览器+代理+养号问题（阶段4b）

**交付物：** 内容生产后能半自动发布到主要平台

---

### 阶段5：转化系统（让流量"变客户"）

**做什么：** 客户来了能接住。

| Skill | 角色 | 功能 | 依赖 |
|-------|------|------|------|
| `lead-collector` | 转化员 | 收集各平台私信/评论 | 平台API |
| `auto-responder` | 转化员 | AI自动回复+引导加微 | AI + CTA话术库 |
| `crm-manager` | 转化员 | 客户线索管理 | 飞书表/企业微信 |
| `solution-packager` | 销售 | 按客户需求打包方案 | 阶段0（业务线详情） |
| `proposal-generator` | 销售 | 自动生成报价单/方案书 | 模板 |

**交付物：** 客户从评论→加微→需求确认→方案→成交的半自动流程

---

### 阶段6：运营中台（让公司"管得住"）

**做什么：** 内部管理数字化。

| Skill | 功能 | 依赖 |
|-------|------|------|
| `daily-data-monitor` | 业务数据日报（客户单量/仓库承载） | 等彭涛给数据库API |
| `financial-tracker` | 财务收支跟踪 | 飞书表 |
| `task-supervisor` | 任务督办（谁做了什么、什么超期） | 飞书督办表（已建） |
| `weekly-report` | 自动生成周报 | 各系统数据汇总 |
| `client-onboarding` | 新客户上线SOP | 标准化流程 |
| `training-material-gen` | 培训材料自动生成 | 知识库 |

**交付物：** 公司日常运营的数据看板+自动化管理

## 总量统计

| 阶段 | Skill数量 | 预计工期 | 依赖 |
|------|-----------|---------|------|
| 0. 公司知识库 | 1 | 2天 | 无 |
| 1. 情报系统 | 4 | 1周 | 阶段0 |
| 2. 内容引擎 | 5 | 1周 | 阶段0+1 |
| 3. 软件工厂 | 6 | 1周 | 阶段0 |
| 4. 分发系统 | 5 | 2周 | 阶段2 |
| 5. 转化系统 | 5 | 2周 | 阶段4 |
| 6. 运营中台 | 6 | 2周 | 阶段0 |
| **总计** | **32个Skill** | **~10周** | |

**阶段0+1+3可以并行**（都只依赖阶段0）。最快路径：

```
周1-2：阶段0（知识库）
周2-3：阶段1（情报）+ 阶段3（软件工厂）并行
周3-4：阶段2（内容引擎）
周5-6：阶段4（分发）
周7-8：阶段5（转化）
周9-10：阶段6（运营中台）
```

## 每个Skill的标准结构

```
skills/
└── skill-name/
    ├── SKILL.md          ← 技能定义（角色身份+目标+约束+SOP）
    ├── config.yaml       ← 配置（API Key引用、参数）
    ├── prompts/          ← 提示词模板
    │   ├── main.md       ← 主提示词
    │   └── ...
    ├── scripts/          ← 自动化脚本（如果需要）
    │   └── ...
    └── tests/            ← 测试用例
        └── ...
```
