# ClawTeam 接入公司 AI 体系落地方案

**日期：** 2026-03-19
**研究对象：** HKUDS/ClawTeam
**定位：** 技术储备方案——公司 AI 体系的运行时编排层设计
**当前状态：** ⚠️ **技术储备阶段，暂不部署。** 先跑通飞书+手动流程，再接入。
**来源整合：** ChatGPT（主结构）+ Claude（商业应用分析+时机判断）+ Gemini（预埋接口建议）

---

## 一、这份文档解决什么问题

我们当前项目已经基本具备三层内容：

1. **公司级认知层**  
   已有 `company-context.md`、`公司AI组织架构与业务模块总盘点.md`，定义了公司背景、业务线、专家池和优先级。

2. **专业能力层**  
   已有或规划中的 Skills，包括市场调研、服务包设计、销售赋能、内容矩阵、文档生成、开发管理等。

3. **业务落地层**  
   已开始沉淀 B 线退货业务的市场增长策略、推广执行方案、销售包、宣传单、素材库规范。

**当前真正缺的，是第 4 层：运行时编排层。**  

也就是说，我们已经知道：
- 公司有哪些业务线
- 每条业务线需要哪些 Agent / Skills
- B 线当前要重点推进什么

但我们还没有一套成熟机制来解决：
- 多个 Agent 如何同时工作
- 谁先做什么、谁依赖谁
- 子 Agent 做完后如何回报
- 如何让一条业务线变成一个真正会运转的“Agent 团队”

ClawTeam 解决的就是这一层。

## 二、ClawTeam 主要做什么

根据官方 README，ClawTeam 的核心定位是：

- 让一个 Leader Agent 自动生成多个 Worker Agent
- 每个 Worker 在独立 git worktree + tmux 窗口里工作
- Agent 之间通过任务系统和 inbox 消息机制协作
- 支持任务依赖、状态更新、团队看板、模板化启动整支队伍
- 兼容 Claude Code、Codex、nanobot 等 CLI agent

它不是：

- CRM
- 文档库
- 素材库
- 飞书替代品
- 业务策略判断器

**它更像“多 Agent 调度器 + 协作底座”。**

参考来源：
- GitHub 仓库：https://github.com/HKUDS/ClawTeam
- README：https://raw.githubusercontent.com/HKUDS/ClawTeam/main/README.md

## 三、它对我们项目的核心启发

### 3.1 启发 1：我们现在缺的不是更多 Agent，而是 Agent 之间的协作机制

我们在《公司 AI 组织架构与业务模块总盘点》中已经定义了 20 个公共专家和 6 条业务线。

问题不在于“没有想清楚 Agent 是谁”，而在于：

- `market-research` 做完后，结果怎么交给 `sales-enablement-builder`
- `content-matrix-planner` 出完内容计划后，谁来拆成脚本、图文、宣传单
- `proposal-quote-generator`、`doc-generator`、`knowledge-base-manager` 怎么共享结果

ClawTeam 的价值就在这里：  
**把“专家名单”变成“协作团队”。**

### 3.2 启发 2：业务线可以被设计成一个团队模板，而不是一堆散 Agent

我们原来更像是：

`Rita -> 业务线 CEO -> 调用不同专家`

ClawTeam 的思路更进一步：

`Rita 给目标 -> 业务线 Leader Agent 自动拉起一支小队 -> 小队自动分工并协作`

这意味着：
- B 线可以是一支“退货业务作战队”
- E 线可以是一支“AI 服务方案队”
- D 线可以是一支“跨境仓配研究与报价队”

### 3.3 启发 3：最适合先接入的不是全公司，而是单业务线试点

ClawTeam 很强，但如果一开始就试图让它管理整家公司，会过重。

对我们来说，最合理的路径不是：

`先做公司级大脑自动化`

而是：

`先选一条最明确、最能变现、任务依赖清晰的业务线来跑通`

这个条件下，**B 线退货业务是最优试点。**

## 四、ClawTeam 在我们体系中的正确位置

建议正式定义为：

`认知层 -> 技能层 -> ClawTeam 运行时编排层 -> 飞书/文档/系统等执行与存储层`

具体对应关系如下：

| 层级 | 我们现有内容 | ClawTeam 是否负责 |
|------|-------------|-------------------|
| 认知层 | `company-context.md`、总盘点、业务线文档 | 否 |
| 技能层 | `market-research`、`sales-enablement-builder`、`content-matrix-planner` 等 | 否，调用这些能力 |
| 运行时编排层 | 任务拆分、依赖、消息、并行、回报、看板 | **是，核心位置** |
| 执行与存储层 | 飞书多维表格、Markdown 文档、系统后台、宣传物料、代码仓库 | 否，但驱动这些产出 |

因此，ClawTeam 的定位不是某个 Skill，而是：

**“让多个 Skills 组成一支队伍去做事”的调度器。**

## 五、哪些地方最适合用 ClawTeam

### 场景 1：B 线退货业务销售包与内容生产

这是当前最推荐的试点。

原因：
- 目标明确：让 B 线从日均 1 万单走向 5-8 万单
- 材料已较完整：市场策略、推广方案、销售包、宣传单、素材库规范都已有基础
- 强依赖协作：研究、话术、内容、宣传单、案例卡、系统内推介需要联动
- 高商业价值：跑通后直接影响现金流和利润

### 场景 2：行业情报引擎 / 客户需求发现引擎

公司 context 已明确要做这两个能力。  
ClawTeam 很适合用来组一支持续运行的“研究小队”，负责：
- 政策监控
- 竞品跟踪
- 客户声音收集
- 信息清洗
- 洞察输出

### 场景 3：软件产品交付链

当 C 线或 B 线未来进入较重的软件开发时，ClawTeam 适合编排：
- architect
- backend
- frontend
- tester
- doc

这一块它的原生优势很强，因为它天然适配 git worktree 和 CLI coding agent。

## 六、哪些地方不适合现在用 ClawTeam

以下事项当前不建议用 ClawTeam 主导：

1. **财务与合同决策**  
   这类事项需要 Rita 直接控制，且高风险、低频，不适合早期自动编排。

2. **客户台账、报价底价、敏感数据管理**  
   这些更适合在本地受限文件或飞书受限权限里管理。

3. **仓库现场运营调度**  
   仓库现场涉及大量现实执行变量，不适合一开始就让多 Agent 编排。

4. **全公司级自动化治理**  
   当前阶段容易过重。先做单业务线试点，再逐步扩展。

## 七、建议优先启动的 3 支试点团队

## 7.1 试点团队 A：B 线退货业务作战队（第一优先）

**目标：** 让退货业务的销售包、内容包、转介绍机制、系统内推介形成一条可持续生产链。

### 团队结构

| 角色 | 对应 Agent/Skill | 主要职责 | 落地状态 |
|------|------------------|---------|---------|
| Leader | `B-line-coordinator` | 拆任务、排优先级、汇总结果、向 Rita 汇报 | ❌ 仅架构定义 |
| 研究员 | `market-research` | 竞品、政策、客户痛点、案例数据 | ✅ 已有完整 Skill |
| 销售官 | `sales-enablement-builder` | 话术、FAQ、案例卡、异议处理 | ❌ 仅架构定义 |
| 内容策划 | `content-matrix-planner` | 抖音/小红书/公众号/朋友圈内容矩阵 | ❌ 仅架构定义 |
| 文档官 | `doc-generator` | 一页纸宣传单、合作模式说明、画册文案 | ❌ 仅架构定义 |
| 资产管理员 | `knowledge-base-manager` | 素材沉淀、版本归档、复盘结论沉淀 | ❌ 仅架构定义 |

### 最适合编排的任务链

1. 研究员更新痛点和政策
2. 销售官生成新的说法和 FAQ
3. 内容策划拆成多平台内容
4. 文档官做成宣传单、案例卡、介绍页
5. 资产管理员入飞书素材库
6. Leader 汇总“今天可发、可讲、可用的东西”

### 预期产出

- 每周更新版销售话术
- 每周 5 条抖音脚本
- 每周 3 条小红书图文
- 每周 1 版一页纸宣传单/案例卡更新
- 每周一次“渠道反馈 -> 口径优化”

## 7.2 试点团队 B：市场情报队（第二优先）

**目标：** 跑通每天/每周的行业情报简报机制，为 A/B/D/E 四条业务线持续输送机会与风险信息。

### 团队结构

| 角色 | 对应 Agent/Skill | 主要职责 | 落地状态 |
|------|------------------|---------|---------|
| Leader | `intelligence-coordinator` | 设定本周监控重点，汇总简报 | ❌ 仅架构定义 |
| 政策监控 | `competitor-monitor` / policy tracker | 追踪电商、快递、跨境、AI 政策 | ❌ 仅架构定义 |
| 客户声音 | `agent-reach` + research worker | 搜社群、平台、短视频评论、论坛 | ✅ agent-reach 已可用 |
| 竞品观察 | `competitor-monitor` | 跟踪产品更新、定价变化、宣传动作 | ❌ 仅架构定义 |
| 洞察整理 | `market-research` | 提炼”这件事对我们意味着什么” | ✅ 已有完整 Skill |

### 适合输出的内容

- 每日一页纸简报
- 重大变化即时预警
- 每周业务机会列表
- 每月竞品变化总结

## 7.3 试点团队 C：软件交付队（第三优先）

**目标：** 在未来的 B 线功能迭代或 C 线定制开发中，提高并行开发和交付效率。

这个场景价值高，但不适合先于 B 线业务作战队启动。

## 八、推荐的接入顺序

### Phase 1：先做最小试点，不接代码，只接内容与销售

先让 ClawTeam 跑 B 线“销售包 + 内容包”。

目标不是炫技，而是验证 3 件事：
- 能否稳定拆任务
- 能否减少 Rita 手工协调
- 能否每周稳定产出一批可用内容

### Phase 2：接入飞书内容中台

当 B 线作战队稳定后，再考虑让输出自动进入飞书多维表格：
- 选题入库
- 文案入库
- 状态流转
- 发布台账
- 复盘归档

### Phase 3：接入市场情报队

把每日行业情报、客户声音、竞品动态纳入长期运行。

### Phase 4：接入软件开发链

等 B 线和情报队跑稳后，再把 ClawTeam 用到开发交付链。

## 十、对我们当前项目的具体借鉴建议

### 建议 1：把“业务线 CEO”落成真正的 Team Leader

我们现在文档里有业务线 CEO 的概念，但还偏抽象。  
可以借 ClawTeam 的思路，把每条业务线 CEO 变成一个真正会：
- 拉起 team
- 分配 task
- 接收 inbox
- 汇总 board

的运行角色。

### 建议 2：把“专家池”改成“可调用 Worker 角色库”

我们现在的专家池定义很完整，但偏“静态名单”。  
ClawTeam 的启发是：这些专家不只是介绍页里的定义，而应该成为可以被 Leader 实际调用的 Worker。

### 建议 3：把飞书多维表格定位成“资产层”，不要让它承担调度

飞书适合存：
- 内容
- 话术
- 发布记录
- 复盘数据

但不适合承担：
- 多 Agent 协作
- 任务依赖管理
- Agent 之间消息流

这一层更适合由 ClawTeam 或类似运行时编排器承担。

### 建议 4：先做“半自动编排”，不要追求完全自治

对我们来说，早期最适合的是：

`Leader Agent 自动拆任务 -> Worker 产出 -> Rita 审核关键节点`

而不是：

`全自动生成并自动发布`

这和我们当前“人在环中”的治理原则一致。

## 十一、接入前提与现实注意事项

ClawTeam 官方要求的基础条件包括：
- Python 3.10+
- tmux
- 已可独立运行的 CLI agent（如 Claude Code、Codex）
- agent 命令在 PATH 中可正常调用

因此，接入前要先确认：

1. 本机是否能稳定运行 `tmux`
2. 所选 CLI agent 是否能在 worktree 中正常启动
3. 是否愿意把 B 线试点放进 git worktree + tmux 的工作模式
4. 是否接受“先把 Leader 和 2-4 个 Worker 跑起来”，而不是一步到位

## 十二、我的最终判断

### 可以借的核心

- Leader + Worker 多 Agent 协作模型
- 任务依赖和自动解锁
- Inbox 消息机制
- Git worktree 隔离
- Team Template 团队模板

### 不要照搬的部分

- 不要一开始用它管理全公司
- 不要让它替代飞书素材库或客户台账
- 不要把高风险决策交给自动编排
- 不要先上最复杂的开发链，先从 B 线内容与销售试点开始

### 最适合我们的定位

**ClawTeam = 我们公司 AI 体系的运行时编排层**

不是大脑，不是业务本身，不是素材库，不是 CRM。  
它是让我们已经设计好的 Agent 组织真正动起来的“调度器”。

## 十四、三方整合结论（Claude + ChatGPT + Gemini 统合）

### 14.1 三方一致的共识

1. **ClawTeam 的定位是"运行时编排层"** — 三方完全一致
2. **B 线退货业务是最佳试点** — 三方完全一致
3. **不要一上来管全公司** — 三方完全一致
4. **飞书是资产存储层，不是调度层** — 三方完全一致
![1774089732546](image/2026-03-19-ClawTeam接入公司AI体系落地方案/1774089732546.png)![1774089736717](image/2026-03-19-ClawTeam接入公司AI体系落地方案/1774089736717.png)
### 14.2 时机判断（核心分歧及裁决）

| 方案 | 主张 | 裁决 |
|------|------|------|
| ChatGPT | 现在就可以开始定义 B 线团队模板并试点 | **暂不采纳** — 设计思路正确但时机太早 |
| Gemini | 先飞书闭环跑 B 线，Skills 里预埋接口，后面再部署 | **采纳** — 最务实 |
| Claude | 先手动跑通内容和销售流程，等知道"哪里最耗时"再自动化 | **采纳** — 避免在没有量的时候优化效率 |

**最终裁决：** 当前是技术储备，不部署。触发条件见下方。

### 14.3 接入 ClawTeam 的触发条件

当以下条件**至少满足 2 条**时，启动 ClawTeam 实际部署：

| 条件 | 说明 |
|------|------|
| B 线客户 ≥ 15 家 | 有了量，效率提升才有意义 |
| 飞书素材库手动运行 ≥ 2 周 | 知道哪些步骤最耗时、最适合自动化 |
| 每天手动协调 Agent ≥ 3 次感到吃力 | 真正需要调度器的信号 |
| 开始做大型软件开发项目（需要多人并行） | ClawTeam 的 git worktree 优势发挥 |

### 14.4 现在就该做的准备（Gemini 建议，采纳）

在后续编写每个 Skill 时，预埋以下"协作接口"字段：

| 字段 | 说明 | 为什么现在就加 |
|------|------|------------|
| **前置依赖** | 这个 Skill 启动前需要哪些 Skill 先完成 | 对应 ClawTeam 的 `--blocked-by` |
| **输出格式** | 这个 Skill 产出的文件名、格式、存放路径 | 对应 ClawTeam 的 Worker 输出给 Leader 的交付物 |
| **完成信号** | 怎么判断这个 Skill 完成了 | 对应 ClawTeam 的 `task update --status completed` |
| **下游接收者** | 产出物应该交给谁继续处理 | 对应 ClawTeam 的 `inbox send` |

这样做的好处：**现在不装 ClawTeam 也不影响手动使用；以后要接入时，Skills 天然兼容。**

### 14.5 关于"是否做成智能体产品"

Gemini 暗示可以把这套体系做成一个智能体 SaaS 软件。我的判断：

**现在不考虑，但方向是对的。**

路径应该是：
1. **先内部用** — 自己公司跑通多 Agent 协作（用 ClawTeam 或类似框架）
2. **积累模板** — 跑出"快递网点 AI 部署模板""电商内容生产模板"等可复用的团队模板
3. **再考虑产品化** — 把这些模板和编排能力包装成 E 线 AI 企业服务的"高级包"

也就是说：**先让自己成为"用 AI 管理公司"的成功案例，再把这套方法卖给别人。**
